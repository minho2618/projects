  현재 프로젝트의 인증 및 인가(Authentication & Authorization) 로직을 분석한 결과입니다.

  프로젝트는 Spring Security와 JWT(JSON Web Token)를 결합한 무상태(Stateless) 인증 아키텍처를 사용하고 있으며, 토큰을 브라우저의 보안 쿠키( HttpOnly )에 담아 처리하고 있습니다.

  ## 1. 핵심 컴포넌트 역할 분석

  ### 🔐 UserAuth.java (사용자 엔티티)

  • Spring Security의  UserDetails  인터페이스를 직접 구현하여 보안 컨텍스트 내에서 인증 객체로 다루어집니다.
  • 주요 상태(Status):  INACTIVE (이메일 미인증),  ACTIVE (활성화),  REMOVED (삭제됨)
  • 이메일( email )을 계정명( Username )으로 사용하고 패스워드는 암호화하여 저장합니다.

  ### 🛡️ SecurityConfig.java (스프링 시큐리티 설정)                                                                                                                                

  • 세션 정책:  SessionCreationPolicy.STATELESS 를 설정하여 서버 세션을 사용하지 않습니다.
  • CORS 설정: 프론트엔드 개발 서버( http://localhost:5173 )로부터의 요청만 허용하며, 쿠키를 전송받을 수 있도록  allowCredentials(true) 를 활성화했습니다.
  • CSRF: REST API 성격에 맞춰 비활성화( disable ) 처리했습니다.
  • 필터 추가: 스프링 시큐리티의 기본 폼 로그인 필터 앞 단계에 커스텀 필터인  JwtAuthenticationFilter 를 배치하여 요청마다 JWT를 검사하도록 구성했습니다.

  ### 🎫 JwtProvider.java (JWT 관리)

  • JWT 토큰의 생성, 파싱, 만료일 검증을 처리하는 유틸리티 서비스입니다.
  • Access Token: Subject에  userId 를 담고,  email 과  type: access  클레임(Claim)을 가집니다.
  • Refresh Token: Subject에  userId 를 담고,  type: refresh  클레임만 가집니다.

  ### ⚙️ JwtAuthenticationFilter.java (요청 가로채기 필터)                                                                                                                         

  • 매 API 요청 시 브라우저가 전송하는 쿠키에서  accessToken 과  refreshToken 을 추출합니다.
  • 자동 갱신(Silent Refresh) 기능이 필터 내에 내장되어 있습니다:
      1.  accessToken 이 유효한 경우 사용자 정보를  SecurityContextHolder 에 세팅하고 통과시킵니다.
      2.  accessToken 이 만료되어  ExpiredJwtException 이 발생하면, 함께 온  refreshToken 을 검증합니다.
      3. 리프레시 토큰이 유효하며 DB( RefreshTokenRepository )에 저장된 값과 일치한다면, 즉시 새로운 Access Token을 생성해 쿠키로 갱신해주고 해당 요청의 인증 정보를 컨텍스트에
      설정하여 사용자 경험을 해치지 않고 인증을 유지합니다.

  ## 2. 주요 인증 비즈니스 흐름

  ### ① 회원가입 및 이메일 활성화

  1.  /api/auth/signup 으로 가입 요청이 오면 패스워드를 BCrypt로 해시화하여  Status.INACTIVE  상태로 저장합니다.
  2. 가입 직후 이메일로 인증 토큰( AuthToken ) 링크를 전송합니다.
  3. 사용자가 메일 링크( /api/auth/verify )를 클릭하면 토큰의 유효성을 DB에서 조회 및 검증한 뒤, 해당 계정을  Status.ACTIVE  상태로 활성화합니다.

  ### ② 로그인 (일반 로그인 및 카카오 OAuth)

  • 일반 로그인 ( /api/auth/login ):
      •  AuthenticationManager 를 통해 이메일/패스워드를 검증합니다.
      • 계정이 활성화( ACTIVE ) 상태인지 검사합니다.
      • 검증 성공 시  accessToken (30분 만료) 및  refreshToken (7일 만료)을 생성하고, Refresh Token을 DB에 저장합니다.
      • 발급한 두 토큰은 브라우저 쿠키( HttpOnly ,  Secure ,  SameSite=Strict  등 보안 옵션 적용) 형태로 응답합니다.
  • 카카오 로그인 ( /api/auth/kakao ):
      • 카카오 인증 서버로부터 전달받은  accessCode 를 통해 Kakao Access Token을 획득하고 프로필(이메일) 정보를 조회합니다.
      • 조회한 이메일로 가입된 로컬 계정이 존재하고 활성화되어 있다면, 동일하게 JWT 토큰셋을 생성하여 쿠키와 헤더에 탑재해 리턴합니다.


  ### ③ 로그아웃

  • 클라이언트로부터 수신한 헤더 및 쿠키에서 토큰 정보를 받아  AuthService.logout 을 실행합니다.
  • DB에 기록되어 있던 사용자의  refreshToken  정보를 즉시 삭제하여 해당 리프레시 토큰을 무효화합니다.
  • 이후 클라이언트 측의 쿠키를 만료(MaxAge = 0)시키는 응답 헤더를 추가합니다.

  ## 1. 세션 기반으로 변경 시 동작 흐름

  ### 1단계: 카카오 연동 및 사용자 식별 (기존과 동일)

  1. 사용자가 카카오 로그인을 진행하면, 백엔드는 카카오 인증 서버로부터 인증 코드(Authorization Code)를 받습니다.
  2. 이 코드를 사용해 카카오에 토큰을 요청하고, 토큰으로 사용자 프로필(이메일 등)을 조회합니다.
  3. 조회된 정보를 바탕으로 기존 가입 정보가 있는지 확인(없다면 회원가입 처리)합니다.

  ### 2단계: 인증 정보 저장 및 세션 생성 (변경되는 부분)

  기존에는 JWT 토큰들을 생성하고 쿠키에 넣었지만, 세션 방식에서는 다음과 같이 처리합니다.

    // 1. 조회한 사용자 정보(UserAuth)를 바탕으로 Authentication 객체 생성
    UsernamePasswordAuthenticationToken authenticationToken =
        new UsernamePasswordAuthenticationToken(user, null, user.getAuthorities());

    // 2. Spring Security Context에 인증 정보 설정
    SecurityContextHolder.getContext().setAuthentication(authenticationToken);

    // 3. HttpSession에 Security Context 저장 (스프링 시큐리티가 자동으로 처리)
    httpServletRequest.getSession().setAttribute(
        HttpSessionSecurityContextRepository.SPRING_SECURITY_CONTEXT_KEY,
        SecurityContextHolder.getContext()
    );

  • 위 코드가 실행되면 WAS(Tomcat 등)는 자동으로 내부 메모리에 세션을 생성하고, 클라이언트(브라우저)에게 세션 ID가 담긴  JSESSIONID  쿠키를 발급합니다.

  ### 3단계: 로그인 상태 유지 (변경되는 부분)

  • 이후 브라우저는 모든 API 요청 시 자동으로  JSESSIONID  쿠키를 백엔드로 보냅니다.
  •  JwtAuthenticationFilter  대신 스프링 시큐리티의 기본 필터( SecurityContextHolderFilter  등)가 해당 세션 ID를 조회하여 메모리에서 인증 정보( SecurityContext )를 자동으로
  복원합니다.

  ## 2. 세션 기반 인증으로 전환 시 장단점 (JWT 대비)

  ### 👍 장점

  4. 구현의 단순함: JWT처럼 토큰을 수동으로 파싱하고 만료 시간을 검사하고, 만료 시점에 Refresh Token으로 재발급(Silent Refresh)하는 복잡한 로직이 전부 필요 없어집니다.
  5. 강력한 통제권: 서버에서 특정 세션을 강제로 만료시킬 수 있으므로, 중복 로그인 방지나 관리자 페이지에서의 강제 로그아웃 처리가 매우 쉽습니다.

  ### 👎 단점 및 고려사항 (현재 프로젝트 기준)

  6. CORS 및 쿠키 설정:
      • 현재 프론트엔드( http://localhost:5173 )와 백엔드( http://localhost:8088 )의 포트/도메인이 다릅니다.
      • 세션 쿠키( JSESSIONID )가 크로스 도메인 상황에서 정상적으로 공유되려면, CORS 설정( allowCredentials(true) ) 및 쿠키 설정( SameSite=None ,  Secure=true )을 매우 철저하게
      맞추어야 합니다.
  7. 서버 확장성:
      • 세션 정보는 기본적으로 서버 메모리(WAS)에 저장됩니다. 추후 서버를 2대 이상으로 확장(스케일아웃)하게 될 경우, 유저가 다른 서버로 접속하면 로그인이 풀리는 문제가 생깁니다.
      • 이를 해결하기 위해 Redis 같은 외부 인메모리 DB를 활용한 Spring Session 분산 환경을 별도로 구축해야 하는 비용이 발생합니다.
