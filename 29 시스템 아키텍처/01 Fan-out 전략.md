**Fan-out**은 메시징 시스템이나 데이터 스트리밍 아키텍처에서 하나의 데이터를 여러 목적지로 동시에 전달하는 방식

발행/구독(Pub/Sub) 모델에서 한 명의 생산자가 생성한 메시지를 다수의 소비자에게 전달할 때 사용

시스템의 요구사항과 병목 구간에 따라 크게 두 가지 방식
- 쓰기 시점 Fan-out (Fan-out on Write / Push Model)
	- 데이터가 생성되는 시점에 해당 데이터를 필요로 하는 모든 사용자의 저장소나 피드에 미리 복사해두는 방식
	- **작동 방식:** SNS에서 사용자가 게시물을 올리면, 그 사용자를 팔로우하는 모든 사람의 '뉴스피드' DB에 게시물 ID를 즉시 입력
	- **장점:** 사용자가 자신의 피드를 조회할 때 추가 연산 없이 미리 구성된 데이터를 읽기만 하면 되므로 조회(Read) 속도가 매우 빠름
	- **단점:** 팔로워가 아주 많은 경우, 한 번의 게시물 작성이 수천만 건의 쓰기 작업을 유발하여 시스템에 부하를 줌
- 읽기 시점 Fan-out (Fan-out on Load / Pull Model)
	- 데이터를 중앙 저장소에만 저장하고, 사용자가 조회를 요청할 때 비로소 관련 데이터를 모아서 보여주는 방식
	- **작동 방식:** 사용자가 피드를 새로고침하면, 시스템이 사용자가 팔로우하는 목록을 조회한 뒤 각 팔로워의 최신 게시물을 실시간으로 취합(Aggregation)
	- **장점:** 쓰기 작업이 단순하며 저장 공간을 절약할 수 있음. 팔로워가 아무리 많아도 게시물은 한 번만 저장.
	- **단점:** 조회를 요청할 때마다 복잡한 쿼리가 실행되므로 읽기 성능이 저하될 수 있음

하이브리드 전략
- 최근 대규모 시스템에서는 두 방식의 장점을 결합하여 사용
- **일반 사용자:** 팔로워 수가 적으므로 **쓰기 시점 Fan-out**을 적용하여 읽기 성능을 최적화
- **유명인:** 팔로워가 너무 많으므로 **읽기 시점 Fan-out**을 적용. 유명인의 게시물은 쓰기 시점에 복사하지 않고, 팔로워들이 피드를 조회할 때만 일반 사용자들의 피드 데이터와 유명인의 게시물을 합쳐서 보여줌.

메시지 브로커에서의 Fan-out
- Kafka, RabbitMQ와 같은 메시지 브로커에서 Fan-out은 특정 메시지를 여러 큐(Queue)로 복제하는 메커니즘
- **RabbitMQ:** `fanout` 타입의 Exchange를 사용하여 바인딩된 모든 큐에 메시지를 무조건 복제함.    
- **Kafka:** 동일한 Topic을 여러 Consumer Group이 구독함으로써 각 그룹이 메시지의 복사본을 독립적으로 처리하게 함.

고려 사항
- **데이터 일관성:** 쓰기 시점 Fan-out 사용 시, 원본 게시물이 삭제되거나 수정되었을 때 복제된 모든 데이터를 업데이트해야 하는 일관성 관리 비용이 발생.
- **인프라 비용:** 데이터를 복제하는 횟수만큼 저장 공간과 네트워크 대역폭 소비가 늘어남.
- **지연 시간:** 쓰기 시점 Fan-out은 비동기 처리가 필수적이며, 시스템 부하에 따라 팔로워의 피드에 게시물이 도달하기까지 시간차가 발생할 수 있음.
