## 1. Offset 기반 페이징 (Offset-based Pagination)

전통적인 페이징 방식으로, `LIMIT`와 `OFFSET` 쿼리를 사용하여 특정 페이지를 건너뛰고 데이터를 가져옵니다.

- **작동 원리**: DB에서 전체 데이터를 정렬한 뒤, 앞부분부터 `OFFSET` 개수만큼의 행을 건너뛰고 필요한 개수(`LIMIT`)만큼 읽어옵니다.    
- **SQL 예시**: `SELECT * FROM posts ORDER BY id DESC LIMIT 10 OFFSET 20;` (21번째부터 10개 추출)    

### 장점

- 구현이 매우 간단하고 직관적입니다.    
- 원하는 페이지로의 직접 이동(Jump)이 가능합니다.    

### 단점

- **성능 저하**: 뒤쪽 페이지로 갈수록 `OFFSET` 값이 커지며, DB는 앞선 행들을 모두 읽은 뒤 버려야 하므로 처리 속도가 급격히 느려집니다.    
- **데이터 중복/누락**: 사용자가 데이터를 조회하는 도중에 새로운 데이터가 삽입되거나 삭제되면, 페이지 경계가 밀리면서 동일한 데이터를 다시 보게 되거나 특정 데이터를 건너뛰게 됩니다.

## 2. Cursor 기반 페이징 (Cursor-based Pagination)

사용자가 마지막으로 본 데이터의 식별자(Cursor)를 기준으로 그다음 데이터를 가져오는 방식입니다.

- **작동 원리**: 마지막 데이터의 특정 값(주로 ID나 생성일시)을 기록해두고, 다음 요청 시 이 ID보다 작은 값 10개를 요청합니다.    
- **SQL 예시**: `SELECT * FROM posts WHERE id < [마지막_ID] ORDER BY id DESC LIMIT 10;`    

### 장점

- **고성능**: 인덱스를 태울 수 있어 전체 행을 스캔하지 않으므로, 데이터양이 많아져도 성능이 일정하게 유지됩니다.    
- **데이터 일관성**: 고정된 지점(Cursor)부터 조회를 시작하므로, 실시간으로 데이터가 추가되어도 중복 노출 문제가 거의 발생하지 않습니다.    

### 단점

- **구현 복잡도**: 정렬 조건이 여러 개일 경우 Cursor 값을 구성하기 까다롭습니다.    
- **페이지 점프 불가**: 5페이지로 바로 가기와 같은 기능을 구현할 수 없으며, 무조건 순차적인 탐색만 가능합니다.
